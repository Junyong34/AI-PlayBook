# AI Tool Catalog: MCP / Skill / Plugin

이 문서는 실제 작업에서 유용하게 사용하는 AI 도구를 한 곳에 정리하기 위한 운영용 카탈로그입니다.
단순 링크 모음이 아니라, 각 도구의 특징과 사용법을 빠르게 확인하고 재사용할 수 있도록 구성합니다.

## 문서 목적

- 현재 사용 중이거나 바로 운영에 투입할 수 있는 MCP, Skill, Plugin을 관리합니다.
- 각 도구의 핵심 특징, 사용 목적, 기본 사용법을 짧게 정리합니다.
- 도구별 지원 AI와 도입 상태를 기록해 유지보수 기준으로 활용합니다.

## 분류 기준

| 구분 | 의미 |
| --- | --- |
| MCP | 외부 도구, 메모리, 코드 탐색 기능을 AI 클라이언트에 연결하는 서버 |
| Skill | 특정 작업 방식을 지침화한 실행 규칙 또는 워크플로우 |
| Plugin | 명령어, Skill, MCP, CLI 등을 묶어 제공하는 확장 패키지 |
| CLI | 터미널에서 직접 실행해 리포트, 분석, 자동화를 수행하는 도구 |

## 상태 기준

| --- | --- |

| `deprecated` | 더 이상 사용하지 않거나 대체 예정 |

## 상세 카탈로그

### Session Report

| 항목 | 내용 |
| --- | --- |
| 유형 | Plugin |
| 지원 AI | both |
| 상태 | `active` |

**특징**

- 프로젝트, Skill, 프롬프트 단위의 토큰 사용량을 시각화합니다.
- 기간별 사용량을 확인해 과다 사용 구간을 찾기 쉽습니다.
- 세션 운영 비용을 정기적으로 점검할 때 유용합니다.

**설명**

AI 작업량과 토큰 소비 패턴을 리포트 형태로 확인하는 도구입니다. 장기 세션, 대형 리팩토링, 문서 작업처럼 컨텍스트가 커지는 작업에서 비용과 효율을 점검하는 데 사용합니다.

**사용법**

- `/session-report --30d`로 최근 30일 사용량을 확인합니다.
- 자연어로 “최근 토큰 사용량 분석해줘”처럼 요청해 리포트를 생성합니다.
- 비용이 비정상적으로 큰 세션은 프롬프트, Skill, 작업 단위로 원인을 나눠 확인합니다.

### Claude MD Management

| 항목 | 내용 |
| --- | --- |
| 유형 | Skill Set |
| 지원 AI | claude-only |
| 상태 | `active` |

**특징**

- `CLAUDE.md` 규칙을 정리하고 중복되거나 모호한 지침을 개선합니다.
- 프로젝트 규칙이 누적되면서 생기는 충돌을 점검하기 좋습니다.
- `revise-claude-md`, `claude-md-improver` 계열 명령을 활용합니다.

**설명**

Claude 기반 프로젝트 규칙 파일을 관리하는 Skill Set입니다. 오래된 규칙, 충돌하는 규칙, 너무 장황한 지침을 정리해 에이전트가 더 일관되게 동작하도록 돕습니다.

**사용법**

- `CLAUDE.md`를 수정하기 전 현재 규칙의 중복과 충돌을 점검합니다.
- `revise-claude-md`로 구조를 정리합니다.
- `claude-md-improver`로 문장과 실행 지침을 개선합니다.

### Serena MCP

| 항목 | 내용 |
| --- | --- |
| 유형 | MCP |
| 지원 AI | both |
| 상태 | `active` |

**특징**

- 의미 기반 코드 검색과 탐색을 지원합니다.
- 단순 문자열 검색보다 코드 구조와 맥락을 파악하는 데 강점이 있습니다.
- 큰 코드베이스에서 관련 함수, 클래스, 호출 흐름을 찾을 때 유용합니다.

**설명**

AI 클라이언트에 코드 이해용 MCP 서버를 연결해 코드 탐색 품질을 높이는 도구입니다. 파일명을 모르는 상태에서 기능의 위치를 찾거나, 특정 동작이 어디서 구현되는지 추적할 때 사용합니다.

**사용법**

- `uv` 기반으로 Serena MCP 서버를 연결합니다.
- 코드 탐색이 필요한 작업에서 MCP 호출을 통해 관련 심볼과 파일을 찾습니다.
- 구조를 파악한 뒤 실제 수정은 로컬 파일 편집 흐름으로 진행합니다.

### Hookify

| 항목 | 내용 |
| --- | --- |
| 유형 | Plugin |
| 지원 AI | both |
| 상태 | `active` |

**특징**

- 반복 실수 패턴을 등록해 작업 전에 차단합니다.
- 팀 또는 개인의 자주 발생하는 실수를 규칙화할 수 있습니다.
- 리뷰 이전 단계에서 품질 게이트 역할을 합니다.

**설명**

자주 반복되는 실수나 금지 패턴을 hook 형태로 관리하는 도구입니다. 예를 들어 특정 명령 실행 금지, 파일 수정 전 체크리스트 확인, 규칙 누락 방지 같은 흐름에 사용할 수 있습니다.

**사용법**

- `hookify`로 반복 실수 패턴을 등록합니다.
- 프로젝트별 금지 명령, 필수 확인 항목, 리뷰 전 체크 항목을 설정합니다.
- 작업 전후로 hook이 의도대로 동작하는지 확인하고 필요하면 패턴을 갱신합니다.

### codeburn

| 항목 | 내용 |
| --- | --- |
| 유형 | Plugin/CLI |
| 지원 AI | both |
| 상태 | `active` |

**특징**

- 토큰 사용량과 세션 비용을 CLI 기반으로 분석합니다.
- 비용이 큰 세션과 비효율적인 작업 흐름을 찾는 데 유용합니다.
- 리포트 기반으로 프롬프트와 작업 분할 방식을 개선할 수 있습니다.

**설명**

AI 사용 비용과 토큰 소모를 가시화하는 분석 도구입니다. Session Report와 함께 사용하면 전체 비용 흐름과 세부 세션 단위 문제를 나눠 확인할 수 있습니다.

**사용법**

- CLI로 리포트를 실행해 세션별 토큰 사용량을 확인합니다.
- 비정상적으로 큰 세션은 작업 범위, 프롬프트 길이, 반복 읽기 여부를 점검합니다.
- 주기적으로 리포트를 저장해 사용량 추이를 비교합니다.

### andrej-karpathy-skills

| 항목 | 내용 |
| --- | --- |
| 유형 | Skill Pack |
| 지원 AI | claude-only |
| 상태 | `active` |

**특징**

- Karpathy식 코딩 원칙을 Skill 형태로 적용합니다.
- 단순성, 목표 중심 실행, 빠른 실험을 강조합니다.
- 과도한 추상화나 장황한 설계를 줄이는 데 도움이 됩니다.

**설명**

코드를 더 단순하고 직접적으로 작성하도록 유도하는 Skill Pack입니다. 복잡한 설계보다 작게 만들고 빠르게 검증하는 방식이 필요한 작업에 적합합니다.

**사용법**

- Plugin 또는 Skill Pack을 설치합니다.
- 구현 전 “단순한 접근 우선” 원칙을 적용할 작업에 활성화합니다.
- 리뷰 시 불필요한 추상화, 과한 일반화, 목표와 무관한 코드를 점검합니다.

### mattpocock/skills

| 항목 | 내용 |
| --- | --- |
| 유형 | Skill Pack |
| 지원 AI | both |
| 상태 | `active` |
| 저장소 | <https://github.com/mattpocock/skills> |

**특징**

- Matt Pocock(Total TypeScript 창립자)이 만든 AI 코딩 에이전트용 Skill 모음집입니다.
- 저장소 제목은 “Skills For Real Engineers”이며, vibe coding이 아닌 실제 엔지니어링을 위한 스킬을 표방합니다.
- GSD, BMAD, Spec-Kit처럼 프로세스를 통째로 소유하는 방식보다 작고 조합 가능한 도구 상자 방식을 지향합니다.
- Claude, Codex 등 특정 모델에 종속되지 않는 model-agnostic 워크플로우를 목표로 합니다.

**핵심 철학**

| 원칙 | 의미 |
| --- | --- |
| Small | 각 스킬을 작게 유지합니다. |
| Easy to adapt | 자기 프로젝트에 맞게 쉽게 수정할 수 있습니다. |
| Composable | 다른 스킬과 조합할 수 있습니다. |
| Model-agnostic | Claude, Codex 등 어떤 모델에서도 작동하도록 설계합니다. |

**해결하려는 실패 패턴**

| 실패 패턴 | 원인 | 처방 | 관련 스킬 |
| --- | --- | --- | --- |
| 에이전트가 의도와 다른 것을 만듦 | 사용자와 에이전트 간 의사소통 격차 | 작업 시작 전 grilling 세션으로 결정 분기 해소 | `/grill-me`, `/grill-with-docs` |
| 에이전트 응답이 너무 장황함 | 프로젝트 전용 용어와 맥락 공유 부족 | `CONTEXT.md` 공용 용어집과 ADR 유지 | `/grill-with-docs` |
| 코드가 작동하지 않음 | 피드백 루프 부재 | TDD red-green-refactor 루프 적용 | `/tdd`, `/diagnose` |
| 코드베이스가 Ball of Mud가 됨 | 빠른 AI 코딩으로 엔트로피도 빠르게 증가 | 매일 설계에 투자하고 주기적으로 아키텍처 개선 | `/to-prd`, `/zoom-out`, `/improve-codebase-architecture` |

**Engineering 스킬**

| 스킬 | 역할 |
| --- | --- |
| `/diagnose` | 어려운 버그와 성능 회귀를 재현, 최소화, 가설 수립, 계측, 수정, 회귀 테스트 흐름으로 진단합니다. |
| `/grill-with-docs` | 계획을 도메인 모델과 대조해 압박 인터뷰를 진행하고 `CONTEXT.md`와 ADR을 즉시 업데이트합니다. |
| `/triage` | 상태 머신 기반으로 이슈를 분류합니다. |
| `/improve-codebase-architecture` | `CONTEXT.md`와 `docs/adr/`를 기반으로 모듈을 더 깊게 만들 기회를 찾습니다. |
| `/setup-matt-pocock-skills` | 최초 1회 설정으로 이슈 트래커, triage 라벨, 문서 경로를 지정합니다. |
| `/tdd` | 수직 슬라이스 단위로 기능 또는 버그를 red-green-refactor 방식으로 진행합니다. |
| `/to-issues` | 계획이나 PRD를 다른 에이전트가 독립적으로 집어갈 수 있는 GitHub 이슈로 분해합니다. |
| `/to-prd` | 지금까지의 대화 컨텍스트를 PRD로 합성하고 GitHub 이슈로 제출합니다. |
| `/zoom-out` | 에이전트가 시스템 전체 관점에서 코드를 설명하도록 유도합니다. |
| `/prototype` | 일회용 프로토타입을 생성합니다. 상태와 비즈니스 로직은 터미널 앱으로, UI는 한 라우트에서 여러 버전을 토글할 수 있게 만듭니다. |

**Productivity 스킬**

| 스킬 | 역할 |
| --- | --- |
| `/caveman` | 기술 정확성은 유지하면서 커뮤니케이션을 초압축해 토큰 사용량을 줄입니다. |
| `/grill-me` | 결정 트리의 모든 가지가 해소될 때까지 요구사항을 집요하게 인터뷰합니다. |
| `/write-a-skill` | progressive disclosure 구조로 새 Skill을 작성합니다. |

**Misc 스킬**

| 스킬 | 역할 |
| --- | --- |
| `/git-guardrails-claude-code` | `push`, `reset --hard`, `clean` 같은 위험한 git 명령을 Claude Code hook으로 사전 차단합니다. |
| `/migrate-to-shoehorn` | 테스트의 `as` 타입 단언을 `@total-typescript/shoehorn`으로 마이그레이션합니다. |
| `/scaffold-exercises` | 섹션, 문제, 해답, 설명 구조의 연습 디렉터리를 생성합니다. |
| `/setup-pre-commit` | Husky, lint-staged, Prettier, 타입체크, 테스트를 포함한 pre-commit hook을 설정합니다. |

**설치 및 사용법**

- `npx skills@latest add mattpocock/skills`를 실행합니다.
- 설치 중 사용할 스킬과 에이전트(Claude Code, Codex 등)를 선택합니다.
- `/setup-matt-pocock-skills`를 반드시 함께 선택합니다.
- 에이전트에서 `/setup-matt-pocock-skills`를 실행해 이슈 트래커(GitHub, Linear, 로컬 파일), triage 라벨, 문서 저장 경로를 설정합니다.
- 이후 `/grill-me`, `/tdd`, `/diagnose` 같은 슬래시 커맨드로 호출합니다.

**잘 맞는 상황**

- 새 기능이나 변경을 시작하기 전 의도를 정렬해야 할 때 `/grill-me` 또는 `/grill-with-docs`를 사용합니다.
- 복잡한 도메인 프로젝트에서는 `CONTEXT.md`와 ADR로 용어를 통일해 재작업을 줄입니다.
- 버그나 성능 이슈는 `/diagnose`로 체계적으로 진단합니다.
- 기능 구현은 `/tdd`로 피드백 루프를 짧게 유지합니다.
- PRD를 작업 단위로 나눌 때 `/to-prd`와 `/to-issues`로 수직 슬라이스를 만듭니다.
- 며칠에 한 번 `/improve-codebase-architecture`로 아키텍처 정비 기회를 찾습니다.
- git 실수를 줄여야 할 때 `/git-guardrails-claude-code`를 적용합니다.

## 운영 원칙

- `active` 항목은 분기별로 실제 사용 여부와 지원 AI 범위를 재확인합니다.
- `MCP`는 연결 장애나 버전 변경이 발생하면 먼저 동작 검증 후 상태를 갱신합니다.
- `Skill Pack`은 기존 규칙 파일과 충돌하지 않는지 확인한 뒤 적용합니다.
- `Plugin/CLI`는 설치 방식, 주요 명령, 리포트 위치를 별도로 기록합니다.
- 더 이상 사용하지 않는 항목은 삭제하지 말고 `deprecated`로 바꾼 뒤 대체 도구를 남깁니다.
