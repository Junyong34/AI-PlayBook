# GoF 패턴 카탈로그

TypeScript 기준으로 22개 GoF 패턴을 빠르게 판단할 수 있도록 정리했다. 각 패턴은 "언제 쓰는가", "언제 과한가", "관련 패턴"만 기억하면 실무 선택에 충분하다.

## 생성 패턴

### Factory Method
- 언제 쓰는가: 생성 규칙을 호출자 밖으로 밀어내고, 구현체 선택을 캡슐화해야 할 때
- 언제 과한가: 구현체가 한두 개뿐이고 단순 함수 하나로 생성이 끝날 때
- 관련 패턴: Abstract Factory, Strategy

### Abstract Factory
- 언제 쓰는가: 서로 맞물리는 객체 묶음을 한 세트로 교체해야 할 때
- 언제 과한가: 실제로는 한 객체만 교체하면 되고 제품군 일관성이 중요하지 않을 때
- 관련 패턴: Factory Method, Builder

### Builder
- 언제 쓰는가: 생성 단계가 많고 선택 옵션이 많으며 유효한 조합을 통제해야 할 때
- 언제 과한가: 매개변수 몇 개로 끝나는 단순 객체를 만들 때
- 관련 패턴: Abstract Factory, Prototype

### Prototype
- 언제 쓰는가: 초기화 비용이 큰 객체를 복제해 변형하는 편이 쉬울 때
- 언제 과한가: 복제보다 생성이 더 명확하고 안전할 때
- 관련 패턴: Builder, Factory Method

### Singleton
- 언제 쓰는가: 프로세스 전역에서 하나만 존재해야 하는 인프라 객체가 명확할 때
- 언제 과한가: 단순히 접근이 편하다는 이유로 전역 상태를 공유하고 싶을 때
- 관련 패턴: Facade, Factory Method

## 구조 패턴

### Adapter
- 언제 쓰는가: 외부 SDK, 레거시 API, 다른 계층의 인터페이스를 현재 코드에 맞춰야 할 때
- 언제 과한가: 인터페이스 차이가 미미하고 호출 지점 한 곳에서만 변환하면 될 때
- 관련 패턴: Facade, Proxy

### Bridge
- 언제 쓰는가: 추상화와 구현 차원이 각각 독립적으로 늘어날 때
- 언제 과한가: 변하는 축이 사실상 하나뿐일 때
- 관련 패턴: Strategy, Abstract Factory

### Composite
- 언제 쓰는가: 트리 구조에서 개별 요소와 집합 요소를 같은 방식으로 다뤄야 할 때
- 언제 과한가: 계층 구조가 고정이고 단순 순회만 하면 될 때
- 관련 패턴: Decorator, Iterator

### Decorator
- 언제 쓰는가: 기존 객체를 감싸며 로깅, 캐싱, 검증 같은 책임을 조합식으로 붙여야 할 때
- 언제 과한가: 기능 조합이 거의 없고 상속 또는 함수 래핑으로 충분할 때
- 관련 패턴: Strategy, Proxy

### Facade
- 언제 쓰는가: 복잡한 서브시스템을 단순한 진입점 하나로 감싸고 싶을 때
- 언제 과한가: 하위 시스템이 이미 충분히 단순하고 호출자도 적을 때
- 관련 패턴: Adapter, Proxy

### Flyweight
- 언제 쓰는가: 매우 많은 객체가 동일한 내부 상태를 공유할 수 있어 메모리 절감이 중요할 때
- 언제 과한가: 객체 수가 많지 않거나 공유 상태 관리가 더 복잡할 때
- 관련 패턴: Composite, Prototype

### Proxy
- 언제 쓰는가: 실제 객체 앞에 접근 제어, 지연 로딩, 원격 호출, 캐싱을 두고 싶을 때
- 언제 과한가: 단순 래퍼일 뿐 행동 통제가 거의 없을 때
- 관련 패턴: Decorator, Adapter

## 행동 패턴

### Chain of Responsibility
- 언제 쓰는가: 요청을 여러 처리기가 순차적으로 받아 조건에 따라 처리하거나 넘겨야 할 때
- 언제 과한가: 처리 순서가 고정이고 단일 함수 분기로도 충분할 때
- 관련 패턴: Command, Mediator

### Command
- 언제 쓰는가: 요청을 객체화해 큐잉, 로깅, 취소, 재실행을 지원해야 할 때
- 언제 과한가: 실행 시점 제어나 히스토리가 필요 없는 단순 메서드 호출일 때
- 관련 패턴: Strategy, Memento

### Iterator
- 언제 쓰는가: 컬렉션 내부 구조를 숨긴 채 순회 방식을 분리해야 할 때
- 언제 과한가: 기본 배열 이터레이션으로 충분할 때
- 관련 패턴: Composite, Visitor

### Mediator
- 언제 쓰는가: 여러 객체 간 통신 규칙이 복잡해 중앙 조정자가 필요할 때
- 언제 과한가: 상호작용 객체 수가 적고 직접 참조가 더 단순할 때
- 관련 패턴: Observer, Facade

### Memento
- 언제 쓰는가: 상태 스냅샷 저장과 복원이 핵심 요구사항일 때
- 언제 과한가: 필요한 상태가 작고 단순 직렬화로 충분할 때
- 관련 패턴: Command, State

### Observer
- 언제 쓰는가: 상태 변경을 여러 구독자에게 느슨하게 전파해야 할 때
- 언제 과한가: 구독자가 거의 없고 직접 호출이 더 읽기 쉬울 때
- 관련 패턴: Mediator, State

### State
- 언제 쓰는가: 상태별 허용 행동과 전이 규칙이 달라져 분기문이 커질 때
- 언제 과한가: 상태 수가 적고 전이가 거의 변하지 않을 때
- 관련 패턴: Strategy, Memento

### Strategy
- 언제 쓰는가: 같은 목적의 알고리즘이나 정책을 런타임에 바꿔 끼워야 할 때
- 언제 과한가: 알고리즘이 하나뿐이거나 교체 가능성이 약할 때
- 관련 패턴: State, Factory Method

### Template Method
- 언제 쓰는가: 알고리즘의 큰 흐름은 같고 일부 단계만 다를 때
- 언제 과한가: 상속보다 함수 조합이나 훅 함수가 더 단순할 때
- 관련 패턴: Strategy, Factory Method

### Visitor
- 언제 쓰는가: 객체 구조는 안정적인데, 그 위에 새 연산이 자주 추가될 때
- 언제 과한가: 객체 구조 자체가 자주 변하거나 타입 수가 적을 때
- 관련 패턴: Composite, Iterator

## 빠른 선택 힌트

- 생성 경계가 문제면: Factory Method, Abstract Factory, Builder
- 외부 인터페이스 정합성이 문제면: Adapter
- 복잡한 하위 시스템 진입점이 문제면: Facade
- 런타임 정책 교체가 문제면: Strategy
- 상태별 규칙과 전이가 문제면: State
- 부가 기능 조합이 문제면: Decorator
- 요청 처리 파이프라인이 문제면: Chain of Responsibility
