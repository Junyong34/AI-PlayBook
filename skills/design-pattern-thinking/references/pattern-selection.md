# 패턴 선택 기준

디자인 패턴은 정답 목록이 아니라 구조적 압력을 다루는 도구다. 먼저 무엇이 반복해서 문제를 만드는지 식별하고, 패턴이 그 문제를 실제로 줄이는지 판단한다.

## 1. 패턴이 필요한지 먼저 판단

다음 중 두세 가지 이상이 동시에 보이면 패턴 검토 가치가 있다.

- 새 정책이나 구현체가 추가될 때마다 `if` 또는 `switch`가 여러 군데 늘어난다.
- 생성 로직이 복잡해 호출자가 내부 조합 규칙까지 알아야 한다.
- 외부 SDK, 레거시 API, UI 계층이 핵심 로직에 직접 스며든다.
- 상태별 허용 동작이 달라지고, 상태 전이 규칙이 분기문에 퍼져 있다.
- 테스트에서 특정 구현을 분리하거나 대체하기 어렵다.

위 징후가 약하면 패턴보다 단순화가 우선이다.

## 2. 단순 대안 우선 규칙

패턴을 추천하기 전에 아래 대안부터 검토한다.

- 함수 추출로 의도가 드러나는가
- 모듈 분리만으로 책임 경계가 선명해지는가
- 의존성 주입만으로 테스트 대체가 쉬워지는가
- 자료구조 변경으로 분기 수를 줄일 수 있는가
- 설정 객체나 맵 테이블로 정책 선택이 끝나는가

이 대안으로 충분하다면 패턴을 억지로 도입하지 않는다.

## 3. 문제 축으로 후보 좁히기

### 생성 패턴부터 볼 때

- 생성 절차가 복잡하고 호출자마다 중복된다
- 제품군이나 생성 규칙이 자주 바뀐다
- 생성 후 불변성이나 유효성 보장이 중요하다

대표 후보:
- Factory Method
- Abstract Factory
- Builder
- Prototype

### 구조 패턴부터 볼 때

- 외부 인터페이스가 맞지 않아 적응 계층이 필요하다
- 복잡한 하위 시스템을 단순한 진입점으로 감싸야 한다
- 기능을 조합으로 확장해야 한다
- 트리 구조나 계층 구조를 동일한 방식으로 다뤄야 한다

대표 후보:
- Adapter
- Facade
- Decorator
- Composite
- Bridge

### 행동 패턴부터 볼 때

- 상태에 따라 행동이 달라진다
- 알고리즘을 런타임에 바꿔야 한다
- 요청을 여러 단계로 흘려보내야 한다
- 객체 간 협력이 얽혀 있다

대표 후보:
- Strategy
- State
- Chain of Responsibility
- Observer
- Command

## 4. 자주 헷갈리는 패턴 비교

### Strategy vs State

- Strategy는 "같은 목적의 알고리즘을 바꿔 끼우는 것"에 가깝다.
- State는 "현재 상태가 바뀌면 객체의 허용 행동과 전이가 달라지는 것"에 가깝다.
- 선택 기준:
  - 사용자가 정책을 고르거나 설정으로 바꾸면 Strategy 쪽 가능성이 크다.
  - 내부 상태 전이가 핵심이고 상태별 규칙이 중요하면 State 쪽 가능성이 크다.

### Factory Method vs Abstract Factory

- Factory Method는 보통 한 종류의 생성 책임을 감춘다.
- Abstract Factory는 서로 관련된 여러 객체 묶음을 일관되게 생성한다.
- 제품군 전체를 바꿔야 하면 Abstract Factory, 단일 생성 경계만 필요하면 Factory Method부터 본다.

### Adapter vs Facade

- Adapter는 인터페이스 불일치를 맞춘다.
- Facade는 복잡한 서브시스템을 더 단순한 진입점으로 감싼다.
- "호환성 문제"면 Adapter, "사용성 문제"면 Facade에 가깝다.

### Decorator vs Strategy

- Decorator는 기존 객체를 감싸며 책임을 덧붙인다.
- Strategy는 핵심 알고리즘 자체를 바꾼다.
- 부가 기능 조합이면 Decorator, 실행 방식 교체면 Strategy를 우선 본다.

## 5. SOLID를 사용하는 방식

SOLID를 체크리스트처럼 암기하지 말고 설계 압력을 설명하는 근거로 사용한다.

- SRP: 한 모듈이 생성, 정책 선택, I/O, 상태 전이를 동시에 떠안는지 본다.
- OCP: 새 케이스를 추가할 때 기존 조건문을 여러 군데 수정해야 하는지 본다.
- LSP: 하위 타입을 끼웠을 때 예외 규칙이 생기지 않는지 본다.
- ISP: 너무 큰 인터페이스가 불필요한 구현을 강요하는지 본다.
- DIP: 핵심 로직이 구체 SDK나 프레임워크에 직접 묶여 있는지 본다.

## 6. 안티패턴 가드레일

- "언젠가 필요할 수도 있다"만으로 추상화를 늘리지 않는다.
- 패턴 이름을 붙이기 위해 불필요한 클래스 계층을 만들지 않는다.
- 단순 객체 리터럴이나 함수 조합으로 끝나는 문제를 객체 패턴으로 부풀리지 않는다.
- 기존 코드 전체를 갈아엎는 제안을 기본값으로 두지 않는다.
- Singleton을 전역 변수의 예쁜 이름으로 쓰지 않는다.

## 7. 점진적 도입 원칙

패턴이 맞더라도 한 번에 전면 도입하지 않는다.

1. 패턴이 필요한 경계 한 곳을 고른다.
2. 새 인터페이스 또는 추상 계층을 그 경계에만 도입한다.
3. 기존 호출자를 모두 바꾸기보다 어댑터나 얇은 래퍼로 연결한다.
4. 테스트를 추가해 새 경계가 실제로 분기와 결합도를 줄였는지 확인한다.
5. 효과가 검증되면 주변 코드로 확장한다.
