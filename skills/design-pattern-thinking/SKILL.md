# 디자인 패턴 사고

패턴 이름을 먼저 고르지 말고, 현재 구조가 왜 불편한지부터 정의한다. 이 스킬의 목적은 GoF 패턴을 기계적으로 적용하는 것이 아니라, 지금 문제를 가장 낮은 복잡도로 풀 수 있는 구조를 선택하게 만드는 데 있다.

## 핵심 원칙

- 먼저 문제를 한 문장으로 정의한다.
- 패턴이 없어도 되는 더 단순한 대안을 먼저 평가한다.
- 패턴을 추천할 때는 이름보다 적용 이유, 비용, 대안, 점진적 도입 순서를 함께 설명한다.
- 패턴이 과하다고 판단되면 그 사실을 분명히 말하고 최소 수정안을 제안한다.
- 기존 코드 전체를 갈아엎는 제안을 기본값으로 두지 않는다.

## 빠른 진단

아래 항목을 기준으로 지금 패턴이 필요한 상황인지 판단한다.

- 복잡도: 조건문, 타입 분기, 상태 전이가 빠르게 늘어나고 있는가
- 변경 빈도: 새 정책, 구현체, 외부 연동, 상태 종류가 자주 추가되는가
- 책임 분리: 생성, 조합, 상태 전이, I/O가 한 모듈에 뭉쳐 있는가
- 테스트 용이성: Mock, Stub, Fake를 넣기 어렵고 경계가 흐린가
- 팀 적합성: 패턴을 도입했을 때 팀이 유지할 수 있는가

패턴이 필요하지 않다면 먼저 `패턴 없이 해결`을 제안한다. 함수 추출, 모듈 분리, 의존성 주입, 자료구조 정리만으로 충분하면 그것이 기본값이다.

## 작업 순서

1. 현재 문제를 한 문장으로 정의한다.
2. 패턴 없이 해결 가능한 단순 대안을 먼저 검토한다.
3. 패턴이 필요하다면 생성, 구조, 행동 패턴 중 어디에 속하는지 좁힌다.
4. 가까운 후보 패턴이 여러 개라면 차이점과 선택 이유를 비교한다.
5. 유지보수성, 확장성, 테스트 용이성, 학습 비용을 함께 설명한다.
6. 점진적 도입 순서를 제안한다.

## 참조 파일 안내

상황에 따라 필요한 문서만 읽는다.

| 상황 | 읽을 파일 |
| --- | --- |
| 패턴이 필요한지 판단하고 안티패턴을 피해야 할 때 | `references/pattern-selection.md` |
| 22개 GoF 패턴 후보를 빠르게 비교해야 할 때 | `references/gof-catalog.md` |
| TypeScript 중심 예시가 필요할 때 | `references/examples.md` |

특히 Factory, Strategy, State, Adapter, Facade처럼 실무에서 자주 헷갈리는 패턴 비교가 필요하면 `references/pattern-selection.md`와 `references/gof-catalog.md`를 함께 읽는다.

## 응답 규칙

- 패턴 이름만 나열하지 않는다.
- "왜 지금 이 패턴인가"와 "왜 더 단순한 대안이 부족하거나 충분한가"를 반드시 설명한다.
- 미래 확장 가능성만으로 과도한 추상화를 권하지 않는다.
- SOLID는 체크리스트가 아니라 설계 압력을 설명하는 근거로 사용한다.
- TypeScript 예시는 인터페이스 경계, 테스트 대체 가능성, 런타임 교체 지점을 드러내도록 작성한다.

## 출력 형식

항상 아래 구조를 기준으로 답한다.

```markdown
## 상황 분석
- 현재 문제:
- 복잡도 수준:
- 변경 빈도:
- 테스트/유지보수 관점의 핵심 리스크:

## 추천 패턴
- 추천:
- 적용 범위:
- 왜 지금 이 패턴인가:

## 대안 비교
- 더 단순한 대안:
- 왜 부족하거나 충분한가:

## 적용 예시
- 코드/구조 예시:
- 점진적 도입 순서:

## 주의할 트레이드오프
- 추가 복잡도:
- 팀 학습 비용:
```

패턴이 불필요하면 `## 추천 패턴` 대신 `## 패턴 없이 해결`로 바꾸고, 가장 작은 수정 단위를 먼저 제안한다.
