# 개발 방법론 및 개발 스타일 가이드

이 문서는 `mfront-web`에서 작업할 때 사용하는 프로젝트 전용 기준 문서입니다.
이 레포는 기능 범위가 넓고 레거시 패턴이 섞여 있으므로, 좋은 개발 스타일은 "크게 바꾸는 것"보다 "작게 바꾸고, 판단 근거를 드러내고, 검증된 결과만 남기는 것"에 가깝습니다.

우선 목적은 AI 에이전트가 구현 과정에서 실수와 과잉 대응을 줄이도록 기본값을 고정하는 데 있습니다.

## 새 코드 기본값

### MUST

- 작업 시작 전에 요구사항, 영향 범위, 기존 패턴, 검증 포인트를 먼저 확인합니다.
- 해석이 여러 개이거나 전제가 불확실하면 숨기지 말고 드러냅니다. 확신이 없으면 멈추고 질문합니다.
- 구현은 항상 사용자 요청과 직접 연결된 최소 변경을 기본값으로 둡니다.
- 변경한 라인은 요청사항, 버그 원인, 검증 목적 중 하나와 직접 연결되게 유지합니다.
- 완료 판단 전에 근거 있는 검증을 수행하고, 결과로 상태를 설명합니다.
- 리뷰 코멘트는 스타일 취향보다 실제 위험과 최소 수정안을 먼저 말합니다.

### SHOULD

- 작업 순서는 `동작 확보 -> 정확성 보강 -> 필요한 범위의 정리`로 가져갑니다.
- 공통화는 세 번째 실제 반복이 생긴 뒤에 검토합니다.
- 하나의 단위가 너무 많은 이유로 바뀌면 책임을 나눕니다.
- 재시도, 중복 클릭, remount, 재호출에도 로직이 안전한지 확인합니다.
- 여러 단계 작업은 단계별 성공 기준과 확인 방법을 먼저 적어 둡니다.
- 더 단순한 접근이 있으면 먼저 채택하고, 복잡한 구조는 실제 필요가 생길 때만 추가합니다.

### AVOID

- 요청되지 않은 구조 개편이나 광범위한 리팩터링
- "언젠가 쓸 것 같아서" 넣는 추상화, 옵션, 과한 확장성
- 현재 작업과 무관한 인접 코드, 주석, 포맷까지 넓게 손보는 것
- 실제로 일어나지 않는 시나리오를 가정한 과한 분기와 에러 처리
- 검증 없이 "아마 될 것"이라고 완료 처리하는 것
- 기존 설정이 느슨하다는 이유로 새 예외를 자연스럽게 추가하는 것

### 기본 작업 흐름

1. 요구사항, 기존 구현, 영향을 받는 범위를 먼저 확인합니다.
2. 단계별 성공 기준과 검증 방법을 먼저 정합니다.
3. 가장 작은 구현으로 동작을 확보합니다.
4. 정확성, 경계 조건, 재실행 안전성을 보강합니다.
5. 현재 작업과 직접 연결된 범위만 정리합니다.
6. 검증을 실행하고, 남은 리스크와 예외를 공유합니다.

### 리뷰 코멘트 기본 형식

리뷰는 아래 형식을 기본값으로 사용합니다.

```text
원칙: 무엇을 어겼는가
위험: 실제로 어떤 버그나 유지보수 비용이 생기는가
최소 수정안: 가장 작은 수정으로 무엇을 바꿔야 하는가
```

## 기존 코드 예외

- 현재 레포는 시기별 패턴이 섞여 있어 모든 파일이 같은 기준을 만족하지 않습니다.
- 작은 버그 수정이나 긴급 대응에서는 이상적인 구조보다 위험을 줄이는 방향이 우선일 수 있습니다.
- 기존 파일이 거칠어 보여도 현재 작업과 무관한 정리를 넓게 수행하지 않습니다.
- 다만 내가 만든 변경이 새로운 죽은 코드, 중복 코드, 사용하지 않는 import를 만들었다면 그것은 정리합니다.
- 기존 예외가 보인다고 해서 새 코드에 그대로 복제하지는 않습니다. 레거시는 설명하고, 새 코드는 기본값을 지키는 쪽을 우선합니다.
